## 2. 当前实现 vs 目标（核心 Gap）

### 2.1 后端 `runW4` 仍用旧契约

**现状**（`PhysicianAiPipelineService.runW4`）：

```java
// 旧：传 JSON 对象 + 全量 diseaseCatalog
input.put("registerId", registerId);
input.put("structuredRecord", loadStructuredRecord(registerId));
input.put("w3Output", loadW3Output(registerId));
input.put("diseaseCatalog", physicianMapper.selectDiseases(null));
// 通过 invokeWorkflow + 通用 apiKey 调用，失败则 FallbackWorkflowEngine.runW4
```

**目标**（WF-05 Start 节点，**全部 String，snake_case**）：

| Dify Start 变量 | 后端组装来源 |
|-----------------|-------------|
| `register_id` | `registerId` |
| `patient_info_text` | 挂号：`gender/age/科室` 拼文本 |
| `chief_complaint` | 病历 `readme` |
| `present_illness` | 病历 `present` |
| `past_history` | 病历 `history` |
| `allergy_history` | 病历 `allergy` |
| `preliminary_diagnosis_text` | 病历 `preliminaryDiagnosis` |
| `preliminary_diseases_text` | 初步诊断关联疾病，格式 `ICD\|名称\|类目`，一行一条 |
| `check_results_text` | `selectCheckResults` 拼文本 |
| `inspection_results_text` | `selectInspectionResults` 拼文本 |
| `w3_analysis_text` | W3 `clinicalImpression` + `overallAnalysis` |
| `abnormal_indicators_text` | W3 `ai_exam_analysis.abnormal_indicators` 提取异常项 |
| `ai_previsit_summary` | 预问诊 `selectLatestAiConsultation` |
| `doctor_notes` | 病历 `physique` 或预留字段 |

**参考模式**：仿照 `W3DifyInputBuilder` 新建 `W4DifyInputBuilder`（`physician-service/.../ai/W4DifyInputBuilder.java`）。

### 2.2 Dify 调用方式应对齐 W3

W3 已采用：**专用 API Key + blocking + 异常抛出**（非静默 Fallback）。

建议 W4 同样：

1. `DifyAiProperties` 增加 `apiKeyW4`、`isW4WorkflowSwitchOn()`、`resolveW4ApiKey()`（**不回退**通用 `api-key`，与 W3 一致）。
2. `DifyWorkflowClient` 增加 `isW4Enabled()`。
3. `runW4` 重构为：
   - `if (difyClient.isW4Enabled())` → `runW4ViaDify(registerId)`  
   - `else` → `fallbackEngine.runW4(...)` + 日志告警
4. `runW4ViaDify` 使用 `difyClient.runWorkflowBlockingWithApiKey(apiKeyW4, inputs, user, traceId)`，失败抛 `DifyWorkflowException`（与 W3 一致）。

`application.yml` / `.env.example` 需增加：

```yaml
workflow-w4: ${DIFY_WORKFLOW_W4:true}   # 开关：true/1/yes/on
api-key-w4: ${DIFY_API_KEY_W4:}         # W4 Workflow App 专用 Key（app-xxx）
```

### 2.3 End 节点输出 vs 当前 persist/前端

**Dify End 8 字段**（成功/兜底分支相同）：

```
status, registerId, suggestions, fallbackSuggestions,
clinicalSummaryForDoctor, differentialDiagnosis, warningSigns, searchAdvice
```

**现状 `persistW4`** 期望旧字段：

```
primaryDiagnosis, differentialDiagnoses, clinicalAdvice
```

**目标写库规则**（设计文档 §8）：

| status | 行为 |
|--------|------|
| `success` / `empty` | 删旧记录 → 遍历 `suggestions` 写入 `ai_diagnosis_suggestion` |
| `fallback` | **不写** `disease_id`；`fallbackSuggestions` 仅返回前端展示，或写日志表（可选） |

字段映射：

| Dify suggestion 字段 | DB 字段 |
|---------------------|---------|
| `diseaseId` | `disease_id` |
| `diagnosisName` | `disease_name` |
| `recommendIcd` | `recommend_icd` |
| `probability` | `probability` |
| `riskLevel` | `risk_level` |
| `treatmentDirection` | `treatment_direction` |
| `diagnosisBasis` | `diagnosis_basis` |
| `sortOrder` | `sort_order` |

**不要**在 W4 落库时自动 `updateDiagnosis` 写确诊（医生点「保存确诊」才写 `medical_record`）。

新建 `DifyW4OutputMapper`（仿 `DifyW3OutputMapper`）：

- 输入：Dify raw outputs
- 输出：统一 API 契约（见 §4.2）
- 处理 `unwrapSource`（兼容 `output_structured` / 平铺）

### 2.4 前端契约过时

**现状** `W4Output`（`physician.ts`）：

```ts
primaryDiagnosis, differentialDiagnoses, clinicalAdvice
```

**目标** 对齐 Dify + 后端 API：

```ts
export interface W4Suggestion {
  diseaseId?: number
  diagnosisName?: string
  recommendIcd?: string
  probability?: number
  riskLevel?: string
  diagnosisBasis?: string
  treatmentDirection?: string
  sortOrder?: number
}

export interface W4FallbackSuggestion {
  diagnosisName?: string
  estimatedIcdPrefix?: string
  probability?: number
  riskLevel?: string
  diagnosisBasis?: string
  note?: string
}

export interface W4Output {
  status?: 'success' | 'empty' | 'fallback'
  registerId?: number
  suggestions?: W4Suggestion[]
  fallbackSuggestions?: W4FallbackSuggestion[]
  clinicalSummaryForDoctor?: string
  differentialDiagnosis?: Array<{ diagnosisName?: string; reason?: string }>
  warningSigns?: string[]
  searchAdvice?: string
  workflowRunId?: string
  modelId?: string
}
```

`PhysicianDiagnosisPage.vue` 需：

1. 运行 W4 后展示 `suggestions` 列表（ICD、概率、依据、治疗方向）。
2. `status=fallback` 时展示 `fallbackSuggestions` + `searchAdvice`，提示医生手动搜疾病。
3. 展示 `clinicalSummaryForDoctor`、`differentialDiagnosis`、`warningSigns`。
4. 页面加载时 `GET /physician/ai/diagnosis-suggestions?registerId=` 回填历史推荐。
5. 点击某条推荐 → 填充确诊表单 + 选中 `diseaseIds`（若 `diseaseId` 存在）。
6. 保存确诊仍走现有 `POST /physician/diagnosis`；可选后续加 `is_adopted` 更新（若后端支持）。

`PhysicianWorkspace.vue` 中 W4 展示区同步更新（若仍使用该入口）。

## 7. 编码约束（必须遵守）

1. **最小 diff**：只改 W4 接入相关文件；不要重构 W1/W2/W3。
2. **Follow W3 模式**：专用 Key、blocking、OutputMapper、InputBuilder。
3. **Start 输入全是 String**：不要把 `diseaseCatalog` 全量传给 Dify。
4. **persistW4**：`fallback` 不写 `disease_id`；不自动写最终确诊。
5. **probability**：Dify 可能给 0～1 小数；DB/前端需统一展示（×100 或保留小数，与现有列类型一致）。
6. **错误处理**：Dify 失败时返回明确错误信息（参考 W3 `DifyWorkflowException`），不要静默 Fallback 除非 W4 未配置。
7. **中文注释**：仅非显而易见业务逻辑；代码风格匹配现有 Java/Vue 文件。
