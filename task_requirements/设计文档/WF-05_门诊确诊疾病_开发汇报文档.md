## 1. 项目背景与目标

### 1.1 业务场景

门诊医生在「门诊确诊」环节需要根据病历、初步诊断、检查检验结果，从 ICD/疾病库中为当前就诊推荐 **3～5 个最可能的确诊疾病**，供医生参考后人工确认。

### 1.2 核心目标

| 目标 | 说明 |
|------|------|
| AI 辅助推荐 | 通过 Dify 工作流 W4「诊断可能的疾病」输出结构化确诊建议 |
| 避免幻觉 | LLM **只能从后端检索出的候选疾病中选择**，不得编造 ICD |
| 可写库 | 输出格式可直接映射 `ai_diagnosis_suggestion` 表 |
| 最小部署 | 服务器仅部署 physician-service + Dify，本地继续开发其他微服务 |

### 1.3 不在本阶段范围

- 工作流不直连数据库、不自动写入确诊结果
- 不替代医生最终确诊
- 不把完整 ICD 库作为 LLM 上下文（仅传 30～80 条候选）

## 2. 整体架构

```text
┌─────────────────────────────────────────────────────────────────┐
│  门诊确诊页面（待开发）                                           │
│  physician-service 组装 Start 字符串 → 调用 Dify W4 Workflow      │
└────────────────────────────┬────────────────────────────────────┘
                             │ POST /v1/workflows/run
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│  Dify W4 工作流（自托管 Docker）                                  │
│  整理输入 → LLM 抽检索线索 → HTTP 查疾病库 → LLM 确诊推荐 → 校验   │
└────────────────────────────┬────────────────────────────────────┘
                             │ POST /api/physician/internal/diseases/ai-search
                             │ Authorization: Bearer INTERNAL_AI_TOKEN
                             ▼
┌─────────────────────────────────────────────────────────────────┐
│  ai-catalog-service（云 JAR，8098）                               │
│  DiseaseAiSearchService → PostgreSQL disease 表（ICD-10 医保 2.0）│
└─────────────────────────────────────────────────────────────────┘
```

本地 `physician-service` 负责触发 Dify 与写库；**不再**暴露 internal 检索接口。

### 2.1 关键设计原则

1. **Start 节点全部使用 String**，避免 Dify 1.11.2 解析复杂 JSON 卡死。
2. **3 个 LLM 节点均使用结构化输出**，代码节点负责校验与格式化。
3. **疾病库在后端 PostgreSQL**，Dify 通过 HTTP 节点按需检索。
4. **双 End 节点**（成功 / 兜底）输出字段一致，便于后端统一解析。

## 3. 开发工作清单

### 3.1 后端（ai-catalog-service + physician-service）

| 模块 | 路径/说明 |
|------|-----------|
| 疾病/药品检索（云） | `ai-catalog-service`：`CatalogInternalController`、`DiseaseAiSearchService`、`DrugAiSearchService` |
| 内部鉴权 | `InternalAiProperties`、`InternalAiAuthInterceptor`（仅 ai-catalog-service） |
| 疾病检索 API | `POST /api/physician/internal/diseases/ai-search`（云 :8098） |
| W4 流水线（本地） | `PhysicianAiPipelineService.runW4`、W4 InputBuilder/OutputMapper |

### 3.2 疾病库（ICD-10 医保 2.0）

| 步骤 | 说明 |
|------|------|
| 数据来源 | 国家 ICD-10 医保 2.0 版 PDF（约 4000 页） |
| 解析工具 | Python + pdfplumber，处理合并单元格与换行 |
| 映射字段 | `disease_code` / `disease_name` / `diseaseicd` / `disease_category` |
| 导入方式 | CSV → PostgreSQL staging 表 → `INSERT INTO disease` |
| 规模 | 全量约 3～8 万条诊断码 |

### 3.3 Dify 工作流编排

在 Dify 1.11.2 中完成 W4 工作流搭建，共 **10 个逻辑节点**（含 2 个 End）：

- 2 个代码节点（整理输入、构造请求、格式化候选、校验输出）
- 3 个 LLM 节点（结构化输出）
- 1 个 HTTP 节点（查疾病库）
- 1 个条件分支
- 2 个 End 节点（成功 / 兜底）

### 3.4 部署架构

| 组件 | 部署位置 | 说明 |
|------|----------|------|
| Dify | Docker（同机） | HTTP 节点访问宿主机需用 `172.17.0.1`，非 `127.0.0.1` / `host.docker.internal`（Linux） |
| physician-service | 宿主机 JAR | `/www/wwwroot/cz/xikang/`，与 `.env` 同目录 |
| PostgreSQL | 远程 43.139.102.203:5432 | `disease` 表存 ICD 全量 |

## 4. 工作流详述

### 4.1 流程图

```text
┌──────────┐
│ 1. Start │  register_id, chief_complaint, 病历/检验文本等（均为 String）
└────┬─────┘
     ▼
┌──────────────┐
│ 2. 整理输入   │  代码：清洗文本、拼 clinical_context、提取 prior_icd_prefixes
└────┬─────────┘
     ▼
┌──────────────────────┐
│ 3. 提取疾病检索线索    │  LLM 结构化：diseaseKeywords, icdPrefixes, ...
└────┬─────────────────┘
     ▼
┌──────────────────────┐
│ 4. 构造疾病检索请求    │  代码：组装 JSON body → disease_search_body
└────┬─────────────────┘
     ▼
┌──────────────────────┐
│ 5. HTTP 查询 ICD      │  POST 172.17.0.1:8098/.../internal/diseases/ai-search
└────┬─────────────────┘
     ▼
┌──────────────────────┐
│ 6. 格式化候选诊断      │  代码：解析 body → candidate_diagnoses_text
└────┬─────────────────┘
     ▼
┌──────────────────────┐
│ 7. 条件分支           │  candidate_count > 0 ?
└────┬─────────┬───────┘
     │有候选    │无候选
     ▼         ▼
┌─────────┐ ┌─────────┐
│ 8A LLM  │ │ 8B LLM  │  确诊推荐 / 兜底建议
│ 9A 校验  │ │ 9B 格式化│
│ 10A End │ │ 10B End │
└─────────┘ └─────────┘
```

### 4.2 节点职责速查

| 节点 | 类型 | 输入 | 输出要点 |
|------|------|------|----------|
| 2 整理输入 | Code | Start 各字符串字段 | `clinical_context`, `prior_icd_prefixes`, `register_id` |
| 3 提取检索线索 | LLM | clinical_context | `diseaseKeywords`, `symptomKeywords`, `icdPrefixes` 等 |
| 4 构造检索请求 | Code | 节点 3 + 2 的字段 | `disease_search_body`（JSON 字符串） |
| 5 HTTP | HTTP | disease_search_body | `body`（后端 Result JSON） |
| 6 格式化候选 | Code | HTTP `body` | `candidate_diagnoses_text`, `candidate_count`, `candidate_id_code_map_json` |
| 7 条件分支 | IF | candidate_count | 有候选 → 8A；无候选 → 8B |
| 8A 核心确诊 | LLM | clinical_context + 候选列表 | `diagnosisSuggestions` 等 |
| 9A 校验输出 | Code | 8A 字段 + candidate_map | 过滤编造 ID，`status=success/empty` |
| 8B 兜底 | LLM | clinical_context | `fallbackSuggestions`, `searchAdvice` |
| 9B 格式化兜底 | Code | 8B 字段 | `status=fallback` |
| 10A/10B End | End | 9A/9B | 8 个统一结构化字段 |

### 4.3 End 节点输出字段（10A / 10B 一致）

| 字段 | 类型 | 成功分支 | 兜底分支 |
|------|------|----------|----------|
| status | String | success / empty | fallback |
| registerId | String | 就诊 ID | 就诊 ID |
| suggestions | Array | 可写库推荐（含 diseaseId） | `[]` |
| fallbackSuggestions | Array | `[]` | 非结构化方向建议 |
| clinicalSummaryForDoctor | String | 综合摘要 | 综合摘要 |
| differentialDiagnosis | Array | 鉴别诊断 | `[]` |
| warningSigns | Array | 危险信号 | `[]` |
| searchAdvice | String | `""` | 建议搜索关键词 |

### 4.4 Dify 代码节点参数命名约定（实践总结）

Dify 会将**输入变量名**作为 `main()` 的 keyword 参数传入，必须与 Python 函数签名一致：

| 来源 | 推荐参数名 |
|------|------------|
| LLM 结构化输出（camelCase） | `diagnosisSuggestions`, `body`, `diseaseKeywords` 等 |
| 代码节点 / Start（snake_case） | `register_id`, `candidate_id_code_map_json` |
| HTTP 节点默认输出 | `body`（不是 `disease_search_response`） |

**务必粘贴完整代码**（含 `import` 与 helper 函数），仅改 `main()` 会导致 `NameError`。

## 5. 后端 API 说明

### 5.1 疾病检索接口

```http
POST /api/physician/internal/diseases/ai-search
Authorization: Bearer ${INTERNAL_AI_TOKEN}
Content-Type: application/json
```

**请求体示例：**

```json
{
  "diseaseKeywords": ["肺炎", "支气管炎"],
  "symptomKeywords": ["咳嗽", "发热"],
  "icdPrefixes": ["J18", "J20"],
  "categoryKeywords": ["呼吸系统疾病"],
  "negativeKeywords": ["肿瘤"],
  "limit": 80
}
```

**响应体示例：**

```json
{
  "code": 200,
  "message": "Success",
  "data": [
    {
      "id": 13491,
      "diagnosisCode": "J15.300",
      "diagnosisName": "B族链球菌性肺炎",
      "matchScore": 96
    }
  ]
}
```

### 5.2 HTTP 节点配置（生产环境）

| 配置项 | 值 |
|--------|-----|
| 方法 | POST |
| URL | `http://172.17.0.1:8098/api/physician/internal/diseases/ai-search` |
| Header | `Authorization: Bearer <INTERNAL_AI_TOKEN>` |
| Body | `{{#构造疾病检索请求.disease_search_body#}}` |
| 超时 | 建议 ≥ 30 秒 |

> 路径必须含 **`/internal/`**，与设计文档初稿 `/api/physician/diseases/ai-search` 不同，以代码实现为准。

## 6. 开发过程中遇到的问题与解决

| # | 问题 | 原因 | 解决方案 |
|---|------|------|----------|
| 1 | 服务器 JAR 找不到 `.env` | EnvLoader 未优先读 `user.dir` | 修改 `EnvLoader`，JAR 同目录加载 `.env` |
| 2 | physician-service 启动失败 | Nacos 未部署，`failFast` 注册失败 | `NACOS_DISCOVERY_ENABLED=false` 默认关闭服务发现 |
| 3 | Dify HTTP 连不上后端 | Linux Docker 无 `host.docker.internal` | 使用 docker 网桥网关 `172.17.0.1` |
| 4 | HTTP 方法/路径错误 | 误配 GET、缺少 `/internal/` | 改为 POST + 正确 URL |
| 5 | 代码节点 `unexpected keyword argument` | 输入变量名与 `main()` 参数不一致 | LLM 输出用 camelCase，与 Dify 输入变量对齐 |
| 6 | `as_list is not defined` | 只粘贴了 `main()` 未粘贴 helper | 粘贴完整 Python 代码 |
| 7 | SQL `syntax error at or near "OR"` | MyBatis foreach 产生 `OR OR` | 修复 `searchDiseasesForAi` XML 拼接逻辑 |
| 8 | disease 表无法清空 | 外键 `medical_record_disease` 引用 | 先删关联 / `UPDATE` 置空再 `DELETE` |
| 9 | INSERT 主键冲突 | 序列 `disease_id_seq` 未同步 | 导入后 `setval` 或执行 `007_sync_all_sequences.sql` |

## 7. 测试验收

### 7.1 测试用例与结果

| 用例 | 场景 | 预期 | 实际结果 |
|------|------|------|----------|
| TC-001 | 典型肺炎，信息完整 | success，3～5 条肺炎相关 ICD | ✅ 4 条，首推 J15.600 革兰阴性菌肺炎 |
| TC-002 | 轻症上感，无肺炎证据 | success，不推荐肺炎 | ✅ 无 J18/J15，首推 J02.905 咽喉痛 |
| TC-003 | SLE 风湿免疫 | 原设计测兜底；全量 ICD 后 | ✅ success，M32.900 系统性红斑狼疮 |
| TC-004 | 极简输入（仅主诉头痛） | 不崩溃，有推荐 | ✅ success，G43.000 偏头痛不伴有先兆 |

### 7.2 验收结论

- **工作流端到端**：Start → HTTP → LLM → 校验 → End 全链路打通。
- **9A 校验有效**：LLM 只能选择候选列表中的 `diseaseId` / ICD，编造项被过滤。
- **兜底分支**：全量 ICD 导入后 TC-003 走成功分支；兜底需另设「库中无匹配」用例（TC-003b）专项验证。

## 10. 汇报要点（一页纸摘要）

**做了什么：**

1. 设计并实现 W4「诊断可能的 diseases」Dify 工作流，采用「LLM + 后端 ICD 检索 + 候选校验」架构，避免 ICD 幻觉。
2. 在 physician-service 新增内部疾病检索 API 与 Bearer 鉴权，支持 Dify HTTP 回调。
3. 完成 ICD-10 医保 2.0 全量导入 PostgreSQL（Python PDF 解析 + SQL 导入）。
4. 完成单机最小部署（physician-service + Dify Docker），解决 Nacos、网络、SQL 等多项联调问题。
5. 通过 4 组测试用例验证成功分支与边界输入。

**技术亮点：**

- Start 全 String + LLM 全结构化输出，适配 Dify 1.11.2 稳定性要求。
- 9A 代码节点严格校验 `diseaseId` / ICD 与候选 map 一致，保证可写库。
- 双 End 统一 schema，后端解析简单。
