## 2. 总体架构设计

### 2.1 架构设计原则

| 原则 | 说明 |
| --- | --- |
| 业务不变，AI 嵌入 | 原有 HIS 门诊业务逻辑、数据库主体、角色权限完全不变，AI 作为辅助能力嵌入各业务环节 |
| 高内聚低耦合 | 每个微服务拥有独立的业务边界和数据存储，服务间通过标准接口通信 |
| 单一职责 | 每个服务只负责一个业务领域的功能 |
| 面向接口编程 | 服务之间通过 RESTful API 通信，不暴露内部实现 |
| AI 能力可插拔 | AI 模块独立部署，原有业务在没有 AI 服务时仍可正常运行 |
| 渐进式扩展 | 架构支持后续新增服务而不影响现有服务 |

## 3. 微服务拆分设计

### 3.1 拆分策略

采用**按业务领域拆分**的策略，遵循以下原则：

1. **业务边界清晰**：每个服务对应一个独立的业务领域，拥有独立的数据库表
2. **团队可独立开发**：每个服务可由 1-2 名开发人员独立完成，无代码层面的耦合
3. **AI 与业务解耦**：AI 服务独立于核心业务服务，业务服务可独立运行
4. **部署独立**：每个服务可独立启动、停止、扩缩容

### 3.2 服务清单与职责

#### 3.2.1 基础设施服务

| 服务名称 | 端口 | 职责 | 核心能力 |
| --- | --- | --- | --- |
| `gateway-service` | 8080 | API 网关 | 路由转发、JWT 鉴权、限流熔断、请求日志、统一跨域配置 |
| `auth-service` | 8081 | 认证鉴权 | 用户登录/登出、JWT 签发与校验、角色权限管理、在线用户状态管理 |

#### 3.2.2 核心业务服务

| 服务名称 | 端口 | 职责 | 对应角色 | 核心能力 |
| --- | --- | --- | --- | --- |
| `registration-service` | 8091 | 挂号收费服务 | 挂号收费员 | 窗口挂号、退号、收费结算、退费、费用查询、日结管理 |
| `physician-service` | 8092 | 门诊医生服务 | 门诊医生 | 患者查看、病历管理、检查/检验申请、门诊确诊、处方开立、处置申请、看诊记录 |
| `medtech-service` | 8093 | 医技检查服务 | 检查/检验/处置医生 | 检查管理、检验管理、处置管理、患者录入、结果录入 |
| `pharmacy-service` | 8094 | 药房管理服务 | 药师 | 药品信息管理、发药管理、退药管理、交易记录查看 |

#### 3.2.3 AI 智能服务

| 服务名称 | 端口 | 职责 | 核心能力 |
| --- | --- | --- | --- |
| `ai-gateway-service` | 8100 | AI 服务内部网关 | AI 服务统一入口、Prompt 模板管理、大模型调用封装、调用日志与计费 |
| `ai-triage-service` | 8101 | AI 导诊服务 | 智能导诊、科室推荐、医生推荐、风险评估 |
| `ai-consult-service` | 8102 | AI 问诊服务 | 预问诊多轮对话、问诊摘要生成、病历草稿自动生成 |
| `ai-diagnosis-service` | 8103 | AI 诊断服务 | 辅助诊断（疾病推荐+ICD编码）、检查结果分析、检查方案推荐 |
| `ai-pharmacy-service` | 8104 | AI 药学服务 | 处方审核（药物冲突/过敏/剂量检测）、用药随访计划生成、康复评估 |

### 3.4 各服务数据库表归属

每个服务只能访问自己领域的数据库表，跨域数据通过服务间 HTTP 调用获取。

| 服务 | 拥有的表 | 说明 |
| --- | --- | --- |
| `auth-service` | employee | 员工信息用于登录认证 |
| `registration-service` | register, regist_level, settle_category | 挂号、挂号级别、结算类别 |
| `physician-service` | medical_record, medical_record_disease, disease, check_request, inspection_request, disposal_request, prescription | 病历、疾病、申请单、处方 |
| `medtech-service` | check_request(读写), inspection_request(读写), disposal_request(读写), medical_technology | 检查/检验/处置申请的执行与结果录入，医技项目 |
| `pharmacy-service` | drug_info, prescription(读), ai_follow_up_plan, ai_follow_up_record | 药品信息、处方读取、随访管理 |
| `ai-triage-service` | ai_triage_record | AI 导诊记录 |
| `ai-consult-service` | ai_consultation_record, ai_medical_record_log | AI 问诊与病历生成记录 |
| `ai-diagnosis-service` | ai_exam_suggestion, ai_exam_analysis, ai_diagnosis_suggestion | AI 诊断与检查分析记录 |
| `ai-pharmacy-service` | ai_prescription_review | AI 处方审核记录 |
| 公共表 | department, scheduling | 科室、排班 — 各服务按需只读访问 |

> **说明**：`check_request`、`inspection_request`、`disposal_request` 三张表由 physician-service 创建（申请），由 medtech-service 读写（执行与结果录入），属于典型的跨服务共享表。在本项目中采用"共享表 + 协调约定"的方式简化实现，不引入分布式事务。

### 4.4 大模型调用封装

所有 AI 子服务通过 `ai-gateway-service` 统一调用大模型 API。封装逻辑如下：

```java
// ai-gateway-service 中的统一调用封装（伪代码示意）
public class AiCallService {

    private final ChatClient chatClient;

    public String call(String prompt, Map<String, Object> variables) {
        // 1. 加载 Prompt 模板并注入变量
        // 2. 调用 ChatClient 发送请求
        // 3. 记录调用日志（调用时间、Token 消耗、响应时间）
        // 4. 返回结果
        // 5. 异常时进行重试（最多 2 次）
        // 6. 超时控制（30 秒）
    }
}
```

### 4.5 AI 服务容错策略

由于 AI 服务依赖外部大模型 API，存在网络延迟或不可用的风险，采用以下容错策略：

| 策略 | 说明 |
| --- | --- |
| 降级处理 | AI 服务不可用时，原有业务流程不受影响，仅跳过 AI 辅助功能，页面显示"AI 服务暂时不可用" |
| 超时控制 | 单次 AI 调用超时时间设为 30 秒，超时后降级处理 |
| 重试机制 | AI 调用失败时自动重试 1 次，两次均失败则降级 |
| 异步调用 | 耗时较长的 AI 任务（如检查结果分析、随访计划生成）采用异步方式，不阻塞主业务流程 |
| 结果缓存 | 相同输入的 AI 分析结果进行缓存，避免重复调用（如检查结果分析） |

## 5. 接口通信设计

### 5.1 通信方式总览

| 通信场景 | 通信方式 | 协议 | 说明 |
| --- | --- | --- | --- |
| 前端 → 后端 | RESTful API | HTTP/JSON | 前端通过 Axios 发送 HTTP 请求到网关 |
| 网关 → 业务服务 | RESTful API | HTTP/JSON | 网关通过路由转发到各微服务 |
| 业务服务 → AI 服务 | RESTful API | HTTP/JSON | 业务服务通过 ai-gateway-service 调用 AI 能力 |
| AI 网关 → AI 子服务 | RESTful API | HTTP/JSON | AI 内部网关路由到各 AI 子服务 |
| 服务 → 数据库 | MyBatis | SQL/JDBC | 各服务通过 MyBatis 访问 PostgreSQL |

> **说明**：本系统采用 **HTTP/REST** 作为统一的通信方式，不引入消息队列或 RPC 框架。原因：
> 1. 项目规模适中，HTTP 调用足以满足性能需求
> 2. 降低架构复杂度，减少学习成本和运维成本
> 3. RESTful API 调试方便，可直接用 Postman/Browser 测试
> 4. 对于耗时较长的 AI 任务，前端采用轮询方式获取结果

### 5.2 API 命名规范

```
统一前缀: /api/{服务名}/{资源名}

示例:
  POST   /api/registration/register          # 挂号
  DELETE /api/registration/register/{id}      # 退号
  GET    /api/registration/register/{id}      # 查询挂号信息
  GET    /api/registration/registers          # 查询挂号列表

  POST   /api/physician/medical-record        # 创建病历
  PUT    /api/physician/medical-record/{id}   # 更新病历

  POST   /api/ai/triage/analyze              # AI 导诊分析
  POST   /api/ai/consult/chat                # AI 预问诊对话
  POST   /api/ai/diagnosis/analyze           # AI 诊断分析
  POST   /api/ai/pharmacy/review             # AI 处方审核
```

### 5.3 统一响应格式

```json
{
    "code": 200,
    "message": "操作成功",
    "data": { ... }
}
```

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| code | Integer | 状态码：200-成功，400-参数错误，401-未认证，403-无权限，500-服务异常 |
| message | String | 状态描述信息 |
| data | Object | 业务数据，失败时为 null |

### 5.4 认证与鉴权流程

```
┌────────┐    ①POST/login     ┌──────────┐   ③校验Token    ┌──────────────┐
│        │ ─────────────────▶ │          │ ──────────────▶ │              │

     │                             │
     │  ⑤请求携带Token            │  ⑥网关拦截校验
     │  (Header: Authorization)    │  (解析JWT, 无效则拒绝)
     │                             │
     ▼                             ▼
  后续API请求 ──────────────▶ 路由转发到业务服务
```

**Token 格式：**

```
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjEsInJvbGUiOiJkZW...
```

**JWT Payload 内容：**

```json
{
    "userId": 1,
    "username": "张医生",
    "role": "physician",
    "deptId": 5,
    "exp": 1716451200
}
```

### 5.5 跨域处理

跨域统一在 `gateway-service` 中处理，业务服务无需单独配置：

```
Access-Control-Allow-Origin: http://localhost:5173
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS
Access-Control-Allow-Headers: Authorization, Content-Type
Access-Control-Allow-Credentials: true
```

### 5.6 服务间内部调用

服务之间的 HTTP 调用通过各服务的**内部端口**直接访问，不经过 `gateway-service`。每个服务同时暴露：

- **外部端口**：通过网关对外提供服务
- **内部端口**：供其他微服务内部调用（同一服务不同端口，或通过服务名 + 端口直接访问）

在本项目中，为简化部署，服务间调用直接使用 `http://localhost:{port}` 的方式。

### 6.5 后端服务启动顺序

服务间存在依赖关系，需按以下顺序启动：

```
1. PostgreSQL 数据库      (基础设施)
2. auth-service          (认证服务，其他服务依赖)
3. gateway-service       (API 网关，依赖 auth-service)
4. registration-service  (核心业务)
5. physician-service     (核心业务)
6. medtech-service       (核心业务)
7. pharmacy-service      (核心业务)
8. ai-gateway-service    (AI 网关)
9. ai-triage-service     (AI 子服务)
10. ai-consult-service   (AI 子服务)
11. ai-diagnosis-service (AI 子服务)
12. ai-pharmacy-service  (AI 子服务)
```

> **说明**：开发阶段可以只启动需要调试的服务。例如开发挂号功能时，只需启动 PostgreSQL + auth-service + gateway-service + registration-service，AI 服务和其它业务服务可以不启动。

## 10. 安全设计

### 10.1 认证安全

| 措施 | 说明 |
| --- | --- |
| JWT Token | 用户登录后签发 JWT，前端每次请求携带 Token |
| Token 过期 | Access Token 有效期 2 小时，Refresh Token 有效期 7 天 |
| 密码存储 | 数据库中密码使用 BCrypt 加密存储，不存储明文 |
| 登录限制 | 同一账号同时只允许一个终端登录，新登录会使旧 Token 失效 |

### 10.2 接口安全

| 措施 | 说明 |
| --- | --- |
| 网关鉴权 | 所有 API 请求经过 gateway-service 统一校验 JWT Token |
| 角色权限 | 不同角色只能访问对应模块的接口（挂号员/医生/药师等） |
| 参数校验 | 后端对所有入参进行校验，防止 SQL 注入、XSS 攻击 |
| 限流 | 网关层对 AI 接口进行限流，防止恶意调用导致大模型费用暴涨 |

### 10.3 AI 调用安全

| 措施 | 说明 |
| --- | --- |
| Prompt 注入防护 | 对用户输入进行清洗和转义，防止恶意 Prompt 注入 |
| 输出过滤 | AI 返回结果经过后处理后展示给用户，过滤敏感信息 |
| 调用审计 | 每次 AI 调用记录完整的输入输出日志，便于追溯 |

```
    "status": "UP"
}
```

### 11.2 统一异常码定义

| 异常码 | 含义 | 场景 |
| --- | --- | --- |
| 200 | 成功 | 正常业务响应 |
| 400 | 参数错误 | 请求参数校验失败 |
| 401 | 未认证 | Token 缺失或过期 |
| 403 | 无权限 | 角色无权访问该接口 |
| 404 | 资源不存在 | 查询的数据不存在 |
| 409 | 业务冲突 | 如重复挂号、处方已审核等 |
| 500 | 服务异常 | 服务内部错误 |
| 503 | AI 服务不可用 | 大模型 API 调用失败，业务降级 |
