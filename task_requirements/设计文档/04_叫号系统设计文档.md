# 熙康云医院 · 叫号系统设计文档

> **文档目的**：在动手之前，把"医生叫号 → 候诊大屏更新 → 患者应答进诊室"这条链路的设计方案、数据模型、接口、实施步骤全部讲清楚。
>
> **范围**：医生工作站 / 候诊大屏 / 报到机联动 / 后端状态机 / 数据库迁移 / 实时通信（WebSocket）。

---

## 一、为什么这件事"挺麻烦"

叫号系统本质上不是"再加一个 CRUD"，而是一个**跨多端、有状态、实时同步**的分布式小系统。它要回答 4 个问题：

1. **谁是号源？** —— 来自挂号 + 报到（已实现）
2. **谁来叫号？** —— 医生在医生工作站点"叫下一个"
3. **叫了之后谁知道？** —— 候诊大屏、报到机、患者手机，所有端要**实时**看到"当前叫到 X 号"
4. **患者来了之后呢？** —— 进诊室，号状态推进，下一个号自动可叫

难点集中在第 3 条：**实时**。如果用轮询（前端每 5 秒查一次接口），简单但卡；用 WebSocket 推送，优雅但要新建配置。

**结论：必须用 WebSocket**，否则演示效果差、医生等不起。

---

## 二、要不要新建数据库？

**不需要新建库。** 在现有 `xikang_hospital` 库的 `register` 表上加字段即可。

理由：
- 叫号是 `register` 记录生命周期的一个阶段，**强归属**于挂号记录
- 拆独立表（如 `call_queue`）会导致每次查询都要 JOIN，得不偿失
- 加字段方案，历史挂号记录的新字段为 NULL，不影响旧数据

只需要：
- 加一个 `migrations/027_calling_fields.sql` 迁移脚本
- 同步改 `Register.java` / `RegistrationMapper.xml` 两处

## 三、数据模型设计

### 3.1 `register` 表新增字段

| 字段 | 类型 | 默认值 | 含义 |
|------|------|--------|------|
| `call_status` | `SMALLINT` | `0` | 号的呼叫状态（见下方枚举） |
| `called_time` | `TIMESTAMP` | NULL | 最近一次被叫的时刻（超时定时任务用此字段判断 5 分钟） |
| `answered_time` | `TIMESTAMP` | NULL | 患者应答（进诊室）的时刻 |
| `call_round` | `SMALLINT` | `0` | 已叫次数；`>= 2` 后过号为终态，禁止重叫 |

### 3.2 `call_status` 状态枚举

```
0  未叫     WAITING       已报到，等医生叫号
1  已叫     CALLED        医生刚叫，等患者进诊室
2  已应答   ANSWERED      患者进诊室，看诊中（visit_state 同步变 2）
3  过号     PASSED        叫了没人来，可重叫
```

### 3.3 状态机

```
                 报到成功
                    │
                    ▼
              ┌─────────────┐
              │ 0 未叫      │
              └─────────────┘
                    │ 医生叫号（call_round=1）
                    ▼
              ┌─────────────┐
       ┌───▶ │ 1 已叫      │◀─────────┐
       │     └─────────────┘          │
       │            │                  │
       │   ┌────────┼────────┐         │
       │   │        │        │         │ 医生重叫
       │   ▼        ▼        ▼         │ (仅 call_round<2)
       │ 过号    超时    应答           │
       │ (3)    (3)     (2)            │
       │   │     │        │            │
       │   │     │        ▼            │
       │   │     │   ┌─────────┐       │
       │   │     │   │ 已应答   │       │
       │   │     │   │ 看诊中  │       │
       │   │     │   └─────────┘       │
       │   │     │                     │
       │   └─────┴───┐                  │
       │            ▼                  │
       │  call_round < 2 ? ──── 否 ────┘
       │            │
       │            ▼ 是
       └────────────┘ (回到 1 已叫)

       call_round >= 2 时，过号为终态：
       状态保持 3，医生无法再次叫号，转人工
```

**关键约束**：
- `call_status=2 已应答` 时，必须同时把 `visit_state` 改为 `2 医生接诊`
- `call_status=1 已叫` 时，**不动 `visit_state`**，仍是 `1`（已挂号）
- `call_status=3 过号` 时，`visit_state` 仍是 `1`
- **过号上限 = 2**：`call_round >= 2` 后，状态保持 `3`，医生重叫会被 `CallingService` 拒绝（返回 400「该号已过号 2 次，请转人工处理」）
- 应答超时 5 分钟，由独立定时任务 `CallingTimeoutScheduler` 自动置为过号

## 四、接口设计

### 4.1 医生端接口（physician-service）

| 接口 | 方法 | 路径 | 作用 |
|------|------|------|------|
| 叫下一个 | POST | `/api/physician/call/next` | 自动选下一个号、置 `called` |
| 指定叫号 | POST | `/api/physician/call/{registerId}` | 叫指定号（重叫过号常用） |
| 患者应答 | POST | `/api/physician/call/{registerId}/answer` | 进诊室，置 `answered` |
| 标记过号 | POST | `/api/physician/call/{registerId}/pass` | 没人来，过号 |
| 当前叫号 | GET | `/api/physician/call/current` | 查当前医生名下"已叫未应答"的号 |

**请求示例**：
```
POST /api/physician/call/next
（无 body，医生身份从 PhysicianAuthContext 取）

Response:
{
  "code": 200,
  "data": {
    "registerId": 1023,
    "patientName": "张三",
    "queueNumber": 5,
    "callRound": 1,           // 第几次叫
    "calledTime": "2026-07-02T10:15:00"
  }
}
```

**"叫下一个"的选取规则**（重要）：
- 同一医生 (`employee_id`)、同一天 (`DATE(visit_date)=今天`)
- `check_in_time IS NOT NULL`（已报到）
- `call_status IN (0, 3)`（未叫或过号）
- 按 `check_in_time ASC`（先报到的先叫）
- 取第一条，更新为 `call_status=1, called_time=now, call_round+=1`

### 4.2 大屏 / 报到机查询接口（registration-service）

| 接口 | 方法 | 路径 | 作用 |
|------|------|------|------|
| 科室叫号板 | GET | `/api/registration/calling-board/{departmentId}` | 查某科室"已叫未应答"的所有号 |
| 全院叫号板 | GET | `/api/registration/calling-board` | 全院汇总（管理员大屏用） |

**响应示例**：
```json
{
  "code": 200,
  "data": [
    {
      "departmentName": "内科",
      "doctorName": "李医生",
      "currentCalling": "5",      // 当前叫到的号序
      "patientName": "张三",
      "callRound": 1,
      "calledTime": "2026-07-02T10:15:00",
      "waitingCount": 4           // 后面还有几人在等
    },
    ...
  ]
}
```

## 五、实时通信：SSE 设计

### 5.0 方案变更记录（2026-07-01）

**原方案**：WebSocket（推送）。

**变更原因**：Phase 0 技术验证发现 Spring Cloud Gateway（WebFlux/Netty）无法把 WebSocket Upgrade 请求转发到下游 Servlet（Tomcat）服务，强行升级时抛 `ClassCastException: ResponseFacade cannot be cast to HttpServerResponse`。这是 Spring 框架层面 WebFlux ↔ Servlet 的不兼容，不是配置问题。

**新方案**：**SSE（Server-Sent Events）**。

**为什么 SSE 适合叫号场景**：
- 叫号是**纯单向**推送（服务器→大屏/报到机），患者端无需回传消息
- SSE 走的是普通 HTTP，Spring Cloud Gateway 可透明转发，不存在协议升级问题
- 浏览器原生 `EventSource` API 内置自动重连，比手写 WebSocket 重连逻辑更稳健
- 单连接开销低，足以支撑一科一屏的规模

**放弃的特性**（相比 WebSocket）：
- 双向通信：叫号场景不需要，没有损失
- 二进制传输：叫号事件都是 JSON 文本，没有损失

### 5.1 为什么不轮询

- 轮询：前端每 5 秒查一次 → 演示时医生叫号后大屏要等最多 5 秒才更新 → 体验差
- 轮询：所有候诊大屏都频繁打后端 → 高峰期 N 个诊室同时跑 → 接口压力大
- SSE：医生叫号后服务器**主动推**给所有订阅端 → 秒级、低开销

### 5.2 SSE 端点设计

```
GET /api/registration/calling/stream/department/{departmentId}   订阅某科室叫号事件
GET /api/registration/calling/stream/doctor/{doctorId}           订阅某医生叫号事件（报到机用）
GET /api/registration/calling/stream/global                      全院订阅（管理员大屏用）
```

**响应头**：
```
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
X-Accel-Buffering: no    # 防 Nginx 缓冲，保证事件立即推送
```

### 5.3 消息格式（SSE event）

每条 SSE 消息形如：
```
event: CALLED
data: {"registerId":1023,"patientName":"张三","queueNumber":5,"doctorName":"李医生","departmentName":"内科","calledTime":"2026-07-02T10:15:00"}

```
（注意：data 后两个换行表示一条事件结束）

事件类型（`event:` 字段的值）：
- `CALLED` —— 医生叫号
- `ANSWERED` —— 患者应答
- `PASSED` —— 过号
- `RESET` —— 重置（管理员操作）
- `HEARTBEAT` —— 心跳（每 30 秒一次，防代理超时断开）

### 5.4 后端实现方式

Spring MVC 原生 `SseEmitter`，无需额外依赖：

```
registration-service/
└── controller/
    └── CallingStreamController.java     # 3 个 GET 端点返回 SseEmitter
└── sse/
    └── CallingSubscriberRegistry.java  # 按 topic 维护 SseEmitter 集合
    └── CallingEventBroadcaster.java    # 业务侧调用，按 topic 推送 + 心跳
```

**关键决策**：叫号服务放在 `registration-service`，不新建微服务。
- 理由：叫号强依赖 `register` 表，与报到、爽约同源
- physician-service 通过 Feign 调 registration-service 的内部接口，触发广播

**SseEmitter 关键参数**：
- `new SseEmitter(0L)` —— 永不超时（大屏常驻）；由心跳和客户端断开管理生命周期
- 注册时发一条 `HEARTBEAT` 让前端确认连接建立
- 客户端断开时 `CompletionCallback` 自动从 registry 移除

### 5.5 网关配置

`/api/registration/calling/stream/**` 走现有的 `registration-service` 路由（`Path=/api/registration/**`），**无需新增路由**。

**网关响应超时**：现有 `spring.cloud.gateway.httpclient.response-timeout: 660s` 是单次 HTTP 响应超时，对 SSE 长连接有影响——SSE 通过心跳（30s 一次）持续刷新活跃状态，不会触发 660s 超时。如真出现超时断连，再调长或针对 stream 路径单独配置。

**鉴权**：大屏是公共设备，stream 路径走白名单。`JwtAuthenticationFilter` 白名单加 `/calling/stream/`（子串匹配，与 `/check-in` 同模式）。

### 5.6 前端接入方式

浏览器原生 `EventSource` API，无需第三方库：

```typescript
const es = new EventSource('/api/registration/calling/stream/department/1')
es.addEventListener('CALLED', (e) => {
  const data = JSON.parse(e.data)
  // 渲染"现在叫号"、闪动、TTS
})
es.addEventListener('ANSWERED', ...)
es.addEventListener('PASSED', ...)
es.onerror = () => { /* EventSource 会自动重连，无需手写 */ }
```

**信息密度设计**：
- 顶部"最新叫号"栏：滚动播报最近 5 条叫号（跨科室都能看到）
- 中部"科室卡片墙"：每个科室一张卡片，显示该科室**当前在叫的号**（不是候诊队列，队列太长了）
- 每张卡片显示最多 3 个叫号（应对一科室多医生同时叫号的情况）
- 卡片底部显示"候诊 N 人"作为参考

**反馈机制**：
- 收到 `CALLED` 事件时：对应科室卡片闪动一下 + 顶部跑马灯新加一条 + TTS 播报
- 字体：科室名 20px、号序 40px、医生/患者 14px（兼顾一屏多卡片）
- 卡片墙用 `grid auto-fill` 自适应屏幕宽度，1080P 屏幕能放 6 张卡片

**对比"一科室一屏"方案（已废弃）**：
- 旧方案路由是 `/calling-board/{departmentId}`，订阅 `department:{id}` 流
- 旧方案适合"每个诊室门口一台屏"，焦点单一更醒目
- 当前实现已统一改为全院版，如果未来要恢复"一科室一屏"，可加回 `:departmentId` 路径参数

## 八、关键设计决策清单

| 决策 | 选择 | 理由 |
|------|------|------|
| 新建数据库？ | ❌ 否 | 复用 `register` 表，加 4 个字段 |
| 新建微服务？ | ❌ 否 | 放在 `registration-service`，与报到同源 |
| 实时通信方式？ | **SSE** | WebSocket 在 Gateway↔Servlet 不兼容（已验证），SSE 走普通 HTTP 无障碍 |
| 状态机：叫号是否改 `visit_state`？ | 仅"应答"改 | 叫号不算医生接诊，应答才算 |
| 过号处理 | 允许重叫，**上限 2 次** | `call_status=3` 可回 `1`，但 `call_round >= 2` 后锁定为终态转人工 |
| 应答超时 | **5 分钟**自动过号 | 新增 `CallingTimeoutScheduler` 定时任务 |
| 过号后队列顺序 | **不重排** | 保持 `check_in_time ASC`，避免"过号反而先看"的错觉 |
| 大屏部署粒度 | **一科室一大屏** | 路由 `/calling-board/{departmentId}`，全院汇总大屏作可选项 |
| 大屏需要登录吗？ | ❌ 不需要 | 公共设备，加白名单 |
| 多医生同时叫号会冲突吗？ | 不会 | 每个医生叫自己名下的号，互不干扰 |
| 报到机提示音 | **做** | 首次加载需用户点击启用（绕过浏览器自动播放限制） |

## 九、风险与未决问题

### 9.1 已识别的风险

1. **~~WebSocket 在 Spring Cloud Gateway 下的代理问题~~** ✅ 已确认发生（2026-07-01）
   - **现象**：gateway 转发 WS Upgrade 请求到 registration-service（Servlet/Tomcat）时抛 `ClassCastException: ResponseFacade cannot be cast to HttpServerResponse`
   - **根因**：Spring Cloud Gateway 内部用 `ReactorNettyRequestUpgradeStrategy`（WebFlux）做 WS 升级，与下游 Servlet 容器的 `ResponseFacade` 类型不兼容
   - **结论**：无法通过配置绕开，已切换为 SSE 方案（见第五章）
   - **遗留的 SSE 风险**（更可控）：
     - 网关 `response-timeout: 660s` 对 SSE 长连接的影响 → 用 30s 心跳刷新活跃
     - Nginx/反代可能缓冲 SSE 流 → 加 `X-Accel-Buffering: no` 响应头（当前阶段单机直连 gateway，暂不需要）

2. **多实例下 SSE session 共享**
   - 如果 registration-service 跑多实例，A 实例上的医生叫号，B 实例上的大屏收不到
   - 单实例开发环境无此问题；要上多实例需要 Redis Pub/Sub 转发
   - **当前阶段单实例，先不解决**，文档不展开

### 9.2 业务规则（已确认）

以下规则已经和需求方确认，作为后续实现的基线：

| 规则 | 决策 | 实现位置 |
|------|------|---------|
| **过号上限** | 同一号最多叫 **2 次**，第 2 次仍无人应答则锁定为"过号终态"，转人工处理 | `CallingService` 加校验：`call_round >= 2` 时拒绝再次叫号，状态保持 `3` |
| **应答超时** | 叫号后 **5 分钟** 未应答，定时任务自动置为过号 | 新增 `CallingTimeoutScheduler`，每分钟扫一次 `called_time` 早于 5 分钟且 `call_status=1` 的记录 |
| **大屏部署粒度** | **一台大屏一个科室** | 路由 `/calling-board/{departmentId}`；SSE 订阅 `/api/registration/calling/stream/department/{departmentId}`；全院汇总大屏作为可选项放在 Phase 4 之后 |
| **报到机提示音** | **做**，但需处理浏览器自动播放限制 | 报到机页面首次加载时显示"点击启用语音"，用户点击后才能播放音频；叫到自己时播 TTS 或预录音频 |

### 9.3 过号后的队列处理（已确认）

- 过号的号**不自动重排**到队尾
- 顺序保持 `check_in_time ASC` 不变
- 医生可在工作站手动点"重叫"任意一个 `call_status=3` 的号
- 这样避免"过号的患者反而先看上"的错觉
