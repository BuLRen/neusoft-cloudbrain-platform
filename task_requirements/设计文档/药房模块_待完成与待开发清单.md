## 3. Bug 与安全问题（P0，必须修复）

### 3.1 【前端 Bug】发药人姓名传值错误
- **位置**：`PharmacyWorkspace.vue` `dispenseSelected()` 与 `returnSelected()`
- **现象**：`pharmacistName` 取的是 `authStore.role`（值为字符串 `'pharmacy'`），不是真实姓名
- **后果**：交易流水、处方记录里的"发药人"列全是字符串 `pharmacy`，无法追溯实际药师
- **修复方案**：改为从 `authStore` 或 `userStore` 取真实姓名（如 `realName`）；`pharmacistId` 也要确认类型对齐

### 3.2 【安全】Controller 无鉴权注解
- **位置**：`PharmacyController` 全部 14 个接口
- **现象**：无 `@PreAuthorize`、无角色校验，任何持有 JWT 的角色都能调
- **后果**：患者、医生账号理论上都能调用发药、删除药品接口
- **修复方案**：与 `auth-service` 商定角色权限方案后，给敏感接口加注解或网关层校验

### 3.3 【数据一致性】处方更新用了全字段 UPDATE
- **位置**：`PharmacyService.dispense/returnDrug` 调用 `prescriptionMapper.update`
- **现象**：药房只该改 `dispensation_status/dispensation_time/pharmacist`，实际 SQL 会覆盖 `diagnosis/total_amount` 等医生负责的字段
- **后果**：未来实体加字段、XML 未同步时，可能把医生写的诊断更新为 null
- **修复方案**：改用现成的 `updateDispensationStatus` 精确更新方法

### 3.4 【并发安全】扣库存 SQL 无乐观锁
- **位置**：`DrugInfoMapper.decreaseStock`
- **现象**：`UPDATE drug_info SET stock_quantity = stock_quantity - #{qty} WHERE id = #{id}`，没有 `AND stock_quantity >= #{qty}` 条件
- **后果**：高并发下两个药师同时发同一药品，应用层校验都通过但实际会负库存
- **修复方案**：SQL 加条件 + 判断 affected rows；或引入版本号字段

### 3.5 【事务边界】AI 调用嵌在数据库事务内
- **位置**：`PharmacyService.dispense()` 在 `@Transactional` 内调 `aiPharmacyClient.createFollowUpPlan`
- **现象**：AI 服务慢或超时时，数据库事务和行锁长时间占用
- **后果**：HikariCP 仅 10 连接，AI 抖动可能拖垮连接池
- **修复方案**：AI 调用移到事务提交后（`TransactionSynchronization.afterCommit` 或 `@TransactionalEventListener`）

### 3.6 【数据一致性】批次库存与汇总库存脱钩
- **位置**：`PharmacyService.dispense/returnDrug` 只动 `drug_info.stock_quantity`，不扣 `drug_stock`
- **现象**：批次表 `drug_stock` 只入库不减库，与汇总数长期不一致
- **后果**：近效期批次 FIFO 管理无法实现；盘点对不上账
- **修复方案**：发药时按 FIFO（`expiry_date` 升序）扣减对应批次 `drug_stock.quantity`，不足时跨批次扣减；退药反向加回

### 3.7 【数据完整性】入库 / 盘点未写交易流水
- **位置**：`PharmacyService.drugInbound/updateStock`
- **现象**：实体注释声明 `pharmacy_transaction.type` 支持 5 种（发放/退回/报损/入库/盘点），实际只写入了"发放"和"退回"
- **后果**：流水查询 `type=入库` 永远空结果；审计链路断
- **修复方案**：入库写 `type=入库`、盘点写 `type=盘点`，金额与数量正负按约定统一

### 3.8 【工程】AI 客户端用 `new RestTemplate()`
- **位置**：`AiPharmacyClient`
- **现象**：直接实例化，无连接池、无超时、无负载均衡，Nacos 注册用不上
- **修复方案**：改为 `@Bean RestTemplate`（配合 `@LoadBalanced`），用服务名 `http://ai-pharmacy-service/...` 替代硬编码 host

### 3.9 【路由】药房路由不允许 admin 巡检
- **位置**：`src/app/router/routes.ts` 药房路由 `meta.roles: ['pharmacy']`
- **现象**：管理员无法访问药房页（其他业务端如 physician/medtech 都允许 admin）
- **修复方案**：改为 `roles: ['pharmacy', 'admin']`

### 3.10 【Mock】药房测试账号被禁用
- **位置**：`src/shared/mock/admin.ts` `pharm_sun` 状态 `disabled`
- **现象**：如果登录链路校验 status，测试账号登不进
- **修复方案**：改为 `enabled`，并补充一个正常状态的药房账号

## 4. 待开发功能 - 后端

### 4.1 【P1】发药二次确认与审方标记
- **目标**：在发药前提供"复核"能力，避免误操作
- **接口设计**：
  - `POST /api/pharmacy/dispense/{registerId}/review` — 提交复核标记（可对接 AI 审方结果，返回 `pass/warn/block`）
  - 现有 `PUT /api/pharmacy/dispense/{registerId}` 增加 `reviewPassed` 必填字段
- **业务规则**：库存不足、近效期、配伍禁忌时强制 warn

### 4.2 【P1】近效期批次查询
- **目标**：支持"先出近期批次"和近效期预警
- **接口设计**：`GET /api/pharmacy/inventory/expiring?days=30`
- **返回**：`[{drugId, drugName, batchNumber, expiryDate, quantity, daysRemaining}]`
- **配套**：4.1 的 FIFO 扣减逻辑依赖此数据

### 4.3 【P1】按剂型/分类查询
- **现状**：`GET /drugs?dosageForm=` 后端支持但前端未用
- **扩展**：`DrugInfo` 增补 `category` 字段（抗生素/OTC/中成药/处方药等），支持 `GET /drugs?category=`

### 4.4 【P1】交易流水补全类型与组合查询
- **目标**：解决参数互斥问题，支持组合筛选
- **接口改造**：`GET /transactions?drugId=&type=&startDate=&endDate=` 改为 AND 组合
- **类型枚举固化**：定义常量 `DISPENSE/RETURN/INBOUND/CHECK/DAMAGE`

### 4.5 【P2】库存盘点工作流
- **接口设计**：
  - `POST /api/pharmacy/inventory/{drugId}/check` — 提交盘点数（实际数 vs 账面数）
  - 自动生成 `type=盘点` 流水，差额正负入库
- **UI 配套**：盘点任务列表、差异复核

### 4.6 【P2】发药单实体启用 or 删除
- **决策点**：`Dispensing` 实体是保留还是删除
  - **保留方案**：实现"发药单打印 / 用药指导单生成"，与 `pharmacy_transaction`（账务流水）区分——一个面向患者、一个面向审计
  - **删除方案**：清理 `Dispensing.java`，依赖 `pharmacy_transaction` 即可
- **建议**：本期删除，后续若有"打印用药标签"需求再启

### 4.7 【P2】操作人来源改造
- **目标**：`pharmacistId/pharmacistName` 不再由前端传，从 JWT Token 取
- **改造点**：`PharmacyController` 入参去掉 `pharmacistId/pharmacistName`，Service 从 `SecurityContext` 或网关透传的 `X-User-Id/X-User-Name` Header 取

### 4.8 【P3】药品供应商与采购（远期）
- 完整药库会涉及供应商档案、采购订单、入库验收、发票
- 当前项目定位为教学/演示，**建议不做**，在文档中标注"超出范围"

## 6. AI 联动专项（本项目差异化能力）

> 参照 `04_AI模块详细设计文档.md` 第 3.7 节，AI 药房服务（`ai-pharmacy-service`, 8104）已规划：用药指导生成、随访计划创建、随访记录收集、随访反馈。
> 当前药房模块**只对接了"随访计划创建"一项**，其余未打通。

### 6.1 【P1】用药指导生成与展示
- **AI 侧接口**：`POST /api/ai/pharmacy/guide`（已存在）
- **后端改造**：`pharmacy-service` 新增 `PharmacyService.generateMedicationGuide(prescriptionId)`，透传给 AI 服务
- **前端入口**：
  - 处方详情区加[用药指导]按钮
  - 药品详情抽屉加[生成用药指导]按钮
- **展示**：右侧抽屉渲染 AI 返回的结构化指导（用法、频次、注意事项、饮食建议）

### 6.2 【P1】随访计划查看入口
- **AI 侧接口**：`GET /api/ai/pharmacy/followup/patient/{patientId}`（已存在）
- **前端改造**：处方详情区加[查看随访计划]按钮，展示该患者所有随访计划与当前阶段
- **价值**：药师发药后能看到"随访计划是否创建成功、进行到哪一阶段"，否则 3.5 中 AI 调用失败对药师是黑盒

### 6.3 【P2】随访创建失败的明细与重试
- **现状**：`dispense` 返回 `followUpFailedPrescriptionIds`，前端只 toast 数量
- **改造**：发药结果区展示失败明细表（处方号、失败原因），每行[重试创建]按钮
- **后端配套**：新增 `POST /api/pharmacy/followup/retry` 接口，按 prescriptionId 重试

### 6.4 【P2】随访反馈录入（药师视角）
- **AI 侧接口**：`POST /api/ai/pharmacy/followup/{planId}/feedback`（已存在）
- **场景**：患者回院复诊时，药师补充录入随访结果
- **前端**：随访计划详情页加[录入反馈]按钮，表单遵循 AI 文档第 3.7.2 节输出格式

### 6.5 【实时性】发药后推送至 AI（可选优化）
- **现状**：发药接口同步调 AI，事务内等待
- **远期**：改为消息队列（MQ）异步通知 AI 创建随访，药师端立即返回
- **前提**：项目引入 MQ 中间件（当前架构无）

## 7. 优先级排期与验收标准

### 7.1 优先级矩阵

| 优先级 | 后端 | 前端 | AI 联动 |
|---|---|---|---|
| **P0 必修** | 3.3、3.4、3.5、3.7 | 3.1、3.9、3.10 | — |
| **P1 主流程补全** | 4.1、4.2、4.3、4.4、4.7 | 5.2、5.3、5.4、5.5、5.6、5.7 | 6.1、6.2 |
| **P2 完善** | 4.5、4.6 | 5.8、5.9、5.10 | 6.3、6.4 |
| **P3 远期/不做** | 4.8 | — | 6.5 |

### 7.2 建议迭代节奏

- **迭代 1（P0 修复）**：安全与数据一致性，纯 bug 修复，不改产品形态
- **迭代 2（P1 核心）**：入库 UI、药品详情、近效期、审方确认 —— 把"药库"职责补齐
- **迭代 3（P1 AI）**：用药指导、随访查看 —— 体现项目 AI 差异化
- **迭代 4（P2 完善）**：盘点、分页、失败重试、文档刷新

### 7.3 验收标准（Definition of Done）

每个条目需满足：

1. **代码**：通过 `mvn compile -pl pharmacy-service -am` 与 `npm run type-check`
2. **联调**：前端在 `npm run dev` 下走通完整交互，后端日志无 ERROR
3. **数据**：涉及库存/流水的改动，`drug_info.stock_quantity`、`sum(drug_stock.quantity)`、`sum(pharmacy_transaction.quantity)` 三者一致
4. **回归**：发药 → 退药主流程仍可用，随访计划仍能自动创建
5. **文档**：若改了接口契约，同步更新 `02_API接口文档.md`
