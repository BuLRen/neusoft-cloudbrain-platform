# 患者端小程序页面规划（基于现有项目结构）

## 1. 文档目的

这份文档不是从零幻想一个全新的患者端，而是基于当前仓库里已经存在的三部分内容整理出来的：

- `wechat_code`：微信小程序工程壳，当前还是初始模板，适合承接患者端
- `xikang-hospital-frontend`：现有 Web 前端，已经有比较完整的患者模块
- `xikang-cloud-hospital`：现有后端微服务，已经提供患者侧大部分核心接口

同时我也参考了 `photos` 文件夹里的设计图。设计图给了我们小程序的大致信息架构和视觉方向，但不是每个功能都已经在项目里实现了，所以本文会区分：

- 当前项目里已经有明确代码和接口支撑的功能
- 设计图里展示了，但现阶段仓库里不一定完整具备的功能

## 2. 当前项目已经具备的患者侧能力

结合 Web 患者端页面、前端 API 和后端接口，当前已经比较明确存在的能力有：

- 患者登录、当前登录用户信息、就诊人列表、本人与家属管理
- AI 导诊
- 挂号流程
- 挂号记录查询
- 挂号取消、挂号缴费、报到
- 科室详情、医生和排班查看
- AI 预问诊
- 我的账单、订单详情、逐项支付、整单支付
- 消息中心、未读消息、已读处理
- 电子病历、就诊记录、单次就诊详情
- 长期健康档案
- 我的处方
- 随访管理、随访计划、随访记录、随访沟通、血糖观测相关能力
- 患者余额充值与余额流水

## 3. 推荐的小程序信息架构

结合你 `photos` 里的设计图和我们刚刚确认的方向，我建议患者端小程序优先采用 5 个底部一级入口：

1. 首页
2. 我的就诊
3. 消息中心
4. 医院信息
5. 个人中心

这次的核心调整是：

- 第一页不只是品牌首页，而是以“挂号”为核心任务页
- 第二页不只放“我的挂号”，而是统一承接“我的挂号、病历、报告、处方、缴费记录”等个人就诊资产

这套结构更符合患者的真实使用路径：

- 第一页：我要办事，先挂号
- 第二页：我要查看自己已经挂过号、看过病、拿过哪些结果
- 第三页以后：看消息、看医院介绍、处理个人资料

## 4. 各页面建议内容

下面按“小程序应该有哪些页面、每页放什么内容、这些内容是否已有项目支撑”来整理。

### 4.1 首页

首页建议做成“挂号主入口页”，而不是纯展示页。

参考设计图，它仍然可以保留医院品牌感，但功能上要优先服务挂号主链路。

建议内容：

- 医院名称、搜索框、轮播 Banner
- 当前最近一次挂号或当前待就诊卡片
- 核心快捷入口区
- 门诊服务入口区
- 公告或健康资讯区

建议首页重点放这些入口：

- AI 导诊
- 按科室挂号
- 按疾病挂号
- 预约挂号
- 当日挂号
- 排队叫号
- 候诊查询
- 门诊缴费
- 报告查询
- 就诊人管理

首页的定位建议明确成：

- 默认入口页
- 以挂号和就诊前准备为主
- 兼顾当前就诊提醒和高频服务入口

当前项目已经支撑的内容：

- 挂号入口：有
- 科室、医生、排班：有
- 挂号记录及待缴费状态：有
- 消息和通知：有
- 就诊人管理：有

当前项目暂时不够明确或需要后续确认的内容：

- 首页搜索是否接真实搜索能力
- 健康资讯是否有后台内容源
- 互联网诊疗是否已有独立后端流程
- 检查预约是否要直接复用现有医技数据，还是首版只做“报告查询”

对应现有 Web 页面/能力：

- `PatientOverview`
- `PatientRegistration`
- `PatientDepartmentDetail`
- `PatientPayment`
- `PatientMessages`

### 4.2 我的就诊页

第二个底部 Tab 我建议正式命名为“我的就诊”，不要再叫“挂号信息”。

原因很简单：

- 如果只叫“我的挂号”，后面塞病历、报告、处方会显得范围太窄
- 如果叫“我的就诊”，就能自然容纳挂号、病历、报告、缴费、处方这些连续链路

这个页面建议做成“个人医疗记录中心”。

建议内容：

- 顶部卡片区：最近一次挂号或当前待就诊信息
- 功能分组入口区
- 我的挂号列表
- 病历和报告快捷入口
- 缴费与处方快捷入口

建议页面内的主要模块：

- 我的挂号
- 待缴费
- 报到码
- 电子病历
- 检查报告
- 检验报告
- 处方记录
- 缴费记录
- 随访记录

我的挂号列表建议展示字段：

- 科室名
- 门诊类型或专病门诊名
- 医生姓名与职称
- 就诊日期、时间段
- 院区/楼层/诊区
- 挂号单号
- 就诊人信息
- 当前状态
- 金额信息

建议支持的操作：

- 去缴费
- 查看详情
- 取消挂号
- 查看报到码
- 去预问诊
- 继续预问诊
- 查看预问诊结果
- 查看病历
- 查看报告

当前项目已经支撑的内容：

- 我的挂号列表：有
- 状态筛选：有
- 挂号详情：有
- 取消挂号：有
- 挂号缴费：有
- 报到二维码：有
- AI 预问诊状态衔接：有
- 病历和单次就诊详情：有
- 长期健康档案：有
- 账单与支付：有
- 处方：有

对应现有 Web 页面/能力：

- `PatientRegistration`
- `PatientRecords`
- `PatientVisitRecord`
- `PatientClinicalProfile`
- `PatientPayment`
- `PatientPrescription`
- `PatientPrevisit`

## 8. 当前建议的开发落点

如果你现在就要开始开发，我建议直接按这条产品主线落：

1. 首页主打挂号和就诊前服务
2. 我的就诊承接挂号、病历、报告、缴费、处方
3. 消息中心做通知闭环
4. 医院信息先静态落地
5. 个人中心承接就诊人和个人资料

一句话说，就是把信息架构定成：

- `首页（挂号） / 我的就诊 / 消息中心 / 医院信息 / 个人中心`

## 9. 一句话结论

基于你现在的项目结构，患者端小程序完全可以不是一个空壳，而是可以直接围绕现有的 Web 患者模块和后端接口，优先落地“首页挂号 -> 我的就诊 -> 缴费 -> 预问诊 -> 消息 -> 个人中心”这条主链路。
